# Self-hosted Windows + PostgreSQL runbook

Этот документ описывает рабочий self-hosted сценарий для проекта «Тёмная Каста». Production больше не описывается как Vercel + Neon. Сайт и PostgreSQL запускаются на инфраструктуре оператора.

## Общая схема

- рабочая папка проекта: `E:\temnayakasta120.ru`;
- приложение: Next.js / React / TypeScript;
- база данных: PostgreSQL;
- ORM: Prisma;
- production-сборка запускается локально на сервере;
- реальные секреты, server config, backups, dumps и пользовательские uploads не коммитятся.

## PostgreSQL

1. Установить PostgreSQL для Windows.
2. Проверить, что командные инструменты PostgreSQL доступны из терминала.
3. Создать отдельную production-базу.
4. Создать отдельного пользователя базы с сильным паролем.
5. Не использовать тестового пользователя или seed-пароль в production.
6. Не публиковать строку подключения в GitHub, логах, документации или чате.

## Server configuration

Production-конфигурация создаётся только на сервере. В репозитории может храниться только безопасный шаблон без реальных значений.

Обязательно проверить:

- публичный адрес сайта;
- подключение к PostgreSQL;
- стабильный ключ Server Actions;
- настройки SMTP;
- юридические данные оператора;
- режим доверия к proxy-заголовкам;
- режим строгого HTTPS.

## Prisma

Перед запуском production нужно проверить Prisma schema, сгенерировать Prisma client и применить миграции к новой базе. Seed не запускается автоматически и запрещён против живой базы без отдельного подтверждения владельца.

## Запуск

Безопасный порядок:

1. клонировать репозиторий;
2. создать server configuration вне Git;
3. установить зависимости;
4. проверить окружение;
5. проверить Prisma schema;
6. сгенерировать Prisma client;
7. применить миграции;
8. собрать production build;
9. запустить production server;
10. проверить регистрацию, вход, почту, профиль, посты, комментарии, модерацию, юридические страницы и оба домена.

## SMTP

Для открытия сайта наружу SMTP должен быть настроен. Если SMTP не заполнен, регистрация может создавать код подтверждения в базе, но письмо пользователю не уйдёт.

## Backups and clean

Backups and dumps must live outside public web-served directories and must not be committed.

`clean` можно использовать только для локальных кэшей и сборочных артефактов внутри корня проекта. Он не должен удалять исходный код, Prisma schema, миграции, пользовательские uploads, документы или server configuration.

## Admin and moderation

После запуска нужно вручную проверить доступ к admin/moderation-разделам, роли пользователей, блокировку неподтверждённой почты и защиту опасных actions.

## Что не используется

Docker не является частью текущего запуска. Vercel и Neon не являются текущим production-контуром.
