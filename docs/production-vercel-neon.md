# Production: self-hosted home server

## Текущая схема

- веб-приложение размещается на self-hosted/домашнем сервере оператора;
- рабочая PostgreSQL-база размещается на инфраструктуре оператора;
- основной публичный адрес: `https://temnayakasta120.ru`;
- дополнительный домен: `temnayakasta120.online`; оба домена должны открывать один и тот же сайт после настройки DNS;
- Vercel и Neon не входят в текущий production-контур.

Публичная документация не должна раскрывать приватный IP, домашнюю сеть, пароли, реальные строки подключения или точные внутренние пути. Для публичной и правовой документации используются только домены и общее описание инфраструктуры.

## Переменные окружения

Production-значения хранятся только в файле окружения на сервере. В репозиторий не должны попадать реальные строки подключения к базе, SMTP-пароли, ключи, токены или иные секреты.

Шаблон переменных может быть указан только с пустыми или условными значениями. Реальные значения заполняются вручную на сервере после создания новой PostgreSQL-базы.

## Обработка данных

Основная инфраструктура сайта является self-hosted. Внешние DNS, регистратор домена, SMTP-провайдер, reverse proxy, CDN, антиспам или мониторинг могут обрабатывать технические данные только если они фактически подключены.

Перед публикацией юридических документов нужно вручную сверить фактическую инфраструктуру, состав внешних сервисов, требования к обработке персональных данных и сведения в уведомлении Роскомнадзора.

## Безопасный порядок запуска

1. Клонировать репозиторий на сервер.
2. Создать production-файл окружения только на сервере.
3. Создать новую PostgreSQL-базу и отдельного пользователя БД.
4. Установить зависимости.
5. Проверить переменные окружения.
6. Проверить Prisma schema.
7. Сгенерировать Prisma client.
8. Применить миграции.
9. Собрать production build.
10. Запустить сайт.
11. Проверить оба домена, HTTPS, авторизацию, регистрацию, отправку почты и юридические страницы.

## Проверки после запуска

1. Проверить, что оба домена открывают один и тот же сайт.
2. Проверить, что реальные секреты не попали в GitHub.
3. Проверить, что база доступна только с нужных интерфейсов/пользователей.
4. Проверить, что reverse proxy и доверие к IP-заголовкам включены только при реальной настройке proxy.
5. Проверить, что HTTPS работает стабильно до включения строгих HTTPS-настроек.
6. Вручную перечитать публичные юридические документы перед публикацией.
