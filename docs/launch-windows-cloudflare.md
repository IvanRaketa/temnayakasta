# Архив: запуск Тёмной Касты на Windows 11 в direct-host режиме

> **Legacy/local scenario.** Этот документ сохранён только как архив старого локального сценария.
> Текущий production размещается на Vercel, а PostgreSQL — в Neon. Cloudflare Tunnel и
> Windows-хост не являются текущей production-инфраструктурой. Актуальная схема описана в
> `docs/production-vercel-neon.md`.

Ниже описан прежний direct-host сценарий: сайт запускался с Windows-хоста, а Next.js слушал
`0.0.0.0:3000`. Эти инструкции нельзя использовать как описание текущего production.

Домены:

- `temnayakasta120.ru` — основной canonical-домен.
- `temnayakasta120.online` — дополнительный домен, открывает тот же сайт.

## 1. Что важно для безопасности

Direct-host означает, что порт `3000` доступен напрямую всем, кому открыт сетевой путь к серверу. Это намеренный текущий режим, но он требует внешнего hardening:

- firewall должен разрешать входящие подключения только там, где это действительно нужно;
- DNS для `temnayakasta120.ru` и `temnayakasta120.online` должен указывать на актуальный хост;
- HTTPS лучше завершать на reverse proxy перед Next.js, например nginx/Caddy/IIS ARR;
- rate limits, security headers и server-side validation должны оставаться включёнными в приложении;
- `TRUST_PROXY` должен оставаться `false`, пока перед приложением нет доверенного proxy, который перезаписывает proxy/IP headers;
- `ENABLE_HSTS` можно включать только после стабильного HTTPS для обоих доменов.

## 2. Проверка окружения

Откройте PowerShell в корне проекта:

```powershell
cd C:\temnayakasta120.ru
node -v
npm -v
npm run env:server-action-key
npm run env:check
npm run prisma:generate

```

В реальном `.env` должны быть заполнены минимум:

```dotenv
DATABASE_URL="postgresql://USER:PASSWORD@127.0.0.1:5432/temnaya_kasta?schema=public"
APP_URL="https://temnayakasta120.ru"
NEXT_SERVER_ACTIONS_ENCRYPTION_KEY="<generated by npm run env:server-action-key>"
SITE_DOMAIN_1="temnayakasta120.ru"
SITE_DOMAIN_2="temnayakasta120.online"
TRUST_PROXY="false"
ENABLE_HSTS="false"
SMTP_HOST="..."
SMTP_PORT="587"
SMTP_SECURE="false"
SMTP_USER="..."
SMTP_PASSWORD="..."
SMTP_FROM="Temnaya Kasta <no-reply@temnayakasta120.ru>"
```

Не вставляйте реальные значения `.env` в документацию, issue, чат или код.

## 3. Production-запуск

```powershell
cd C:\temnayakasta120.ru
npm run prisma:generate
npm run env:server-action-key
npm run build
npm run start
```

Скрипт `npm run start` запускает Next.js на `http://0.0.0.0:3000`. Это значит, что процесс слушает все сетевые интерфейсы хоста. Для локальной проверки на сервере:

```powershell
Invoke-WebRequest http://127.0.0.1:3000 -UseBasicParsing
Invoke-WebRequest http://localhost:3000 -UseBasicParsing
```

Для проверки прослушивания:

```powershell
Get-NetTCPConnection -State Listen -LocalPort 3000 |
  Format-Table LocalAddress,LocalPort,OwningProcess
```

Ожидаемо: `0.0.0.0:3000` или `:::3000`.

## 4. Firewall

Проверьте входящие правила Windows Firewall:

```powershell
Get-NetFirewallRule | Where-Object DisplayName -like "*3000*" |
  Format-Table DisplayName,Enabled,Direction,Action,Profile
```

Если сайт должен быть доступен из интернета напрямую, разрешающее правило должно быть узким и понятным. Если доступ должен идти только через будущий reverse proxy, прямой внешний доступ к `3000` нужно закрыть после установки proxy.

Пример проверки порта:

```powershell
Test-NetConnection temnayakasta120.ru -Port 3000
Test-NetConnection temnayakasta120.online -Port 3000
```

Не удаляйте firewall-правила без отдельного подтверждения владельца.

## 5. HTTPS и reverse proxy

Сейчас Next.js может слушать HTTP на `3000`, а публичный HTTPS должен быть отдельным инфраструктурным решением. Рекомендуемый следующий шаг:

- поставить reverse proxy перед Next.js;
- выпустить TLS-сертификаты для `temnayakasta120.ru` и `temnayakasta120.online`;
- проксировать запросы на `http://127.0.0.1:3000` или `http://localhost:3000` уже внутри хоста;
- после этого включить `ENABLE_HSTS="true"`;
- включать `TRUST_PROXY="true"` только если proxy корректно перезаписывает `X-Forwarded-For`, `X-Real-IP` или CDN headers.

В текущем direct-host режиме `TRUST_PROXY="false"` защищает rate limits и аудит от spoofed IP headers.

## 6. Автозапуск сайта после перезагрузки Windows

Создайте задачу Windows Task Scheduler для Next.js:

```powershell
$Action = New-ScheduledTaskAction -Execute "powershell.exe" -Argument "-NoProfile -ExecutionPolicy Bypass -Command `"cd C:\temnayakasta120.ru; npm run start`""
$Trigger = New-ScheduledTaskTrigger -AtStartup
$Principal = New-ScheduledTaskPrincipal -UserId "$env:USERNAME" -LogonType Interactive -RunLevel Highest
Register-ScheduledTask -TaskName "TemnayaKastaNextStart" -Action $Action -Trigger $Trigger -Principal $Principal -Description "Start Temnaya Kasta Next.js production server"
```

Проверка/ручной старт:

```powershell
Start-ScheduledTask -TaskName "TemnayaKastaNextStart"
Get-ScheduledTask -TaskName "TemnayaKastaNextStart"
```

## 7. Резервная копия PostgreSQL

Backups нельзя хранить в публичной директории, отдаваемой сайтом. Не кладите дампы в `public`, не коммитьте их и не отправляйте в чаты.

Создать каталог:

```powershell
New-Item -ItemType Directory -Force C:\temnayakasta120.ru\backups\db
```

Если `pg_dump` доступен в PATH:

```powershell
$Stamp = Get-Date -Format "yyyyMMdd-HHmmss"
$PgDumpUrl = ($env:DATABASE_URL -split "\?")[0]
pg_dump --dbname="$PgDumpUrl" --format=custom --file="C:\temnayakasta120.ru\backups\db\temnaya-kasta-$Stamp.dump"
```

Храните реальные дампы отдельно от production root, в зашифрованном хранилище с retention-политикой. Каталог `backups` уже должен игнорироваться git-ом.

## 8. Резервная копия проекта

Без `node_modules`, `.next`, `backups` и `.env`:

```powershell
$Stamp = Get-Date -Format "yyyyMMdd-HHmmss"
```

Реальный `.env` храните отдельно в менеджере паролей или защищённом архиве.

Команда `npm run clean` удаляет `.next`, временные сборочные каталоги и логи внутри workspace:

```powershell

npm run prisma:generate
npm run env:server-action-key

```

Не удаляйте `node_modules`, `public/uploads`, `prisma/migrations`, `.env` и `backups`, если не понимаете последствия.

## 11. Финальный чеклист

- `npm run env:check` проходит.
- `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` задан до `npm run build`.
- `TRUST_PROXY="false"` в direct-host режиме.
- `ENABLE_HSTS="false"`, пока нет стабильного HTTPS для обоих доменов.
- `npm run prisma:generate` проходит.
- `npm run build` проходит.
- `npm run start` слушает `0.0.0.0:3000`.
- Firewall и внешний доступ к `3000` проверены вручную.
- `npm run smtp:check` проходит и тестовое письмо приходит.
- Регистрация, подтверждение e-mail, вход, создание поста, комментарии, реакции, подписки и жалобы проверены.
- Обычный пользователь не видит админку и модерацию в меню.
- Админ/модератор видят свои служебные разделы.
- Premium без оплаты: статус выдаётся админом, значок/переливание/скрытие рекламы работают.
- `temnayakasta120.ru` и `temnayakasta120.online` открывают один сайт.
- Есть свежий дамп PostgreSQL и архив проекта, но backups не лежат в `public` и не коммитятся.
- Юридические тексты, контакты и вопрос уведомления Роскомнадзора проверены вручную владельцем.

## 12. Future infrastructure

Cloudflare Tunnel, Cloudflare WAF, nginx/Caddy, TLS, firewall hardening и перенос `3000` за reverse proxy требуют отдельного подтверждения владельца. После появления доверенного proxy можно отдельно пересмотреть `TRUST_PROXY`, HSTS и WAF/rate-limit rules.
