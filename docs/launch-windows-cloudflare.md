# Запуск Тёмной Касты на Windows/self-hosted сервере

Этот документ описывает актуальный self-hosted сценарий: приложение Next.js и PostgreSQL размещаются на инфраструктуре владельца проекта. Внешние DNS, SMTP, reverse proxy, CDN или WAF могут использоваться только как отдельные технические сервисы, если они реально подключены.

Домены:

- `temnayakasta120.ru` — основной canonical-домен.
- `temnayakasta120.online` — дополнительный домен, открывает тот же сайт.

## 1. Базовая схема

- Код проекта хранится в GitHub.
- Рабочая папка на сервере: `E:\temnayakasta120.ru`.
- Реальный `.env` хранится только на сервере и не коммитится.
- PostgreSQL работает как self-managed база.
- Дампы, архивы, uploads и секреты не должны попадать в GitHub.

## 2. Безопасный порядок запуска

1. Склонировать свежий репозиторий в рабочую папку.
2. Создать новый реальный `.env` на сервере.
3. Указать строку подключения к новой PostgreSQL-базе.
4. Проверить server-action key, домены, SMTP и юридические переменные.
5. Установить зависимости.
6. Проверить Prisma schema.
7. Применить миграции.
8. Собрать production-сборку.
9. Запустить production-процесс.
10. Проверить сайт локально и через домены.

## 3. Сеть и reverse proxy

Для публичного сайта безопаснее держать Next.js за reverse proxy. Публичный HTTPS должен быть настроен до включения строгих HTTPS-заголовков.

- `TRUST_PROXY` оставлять выключенным, пока нет доверенного proxy.
- HSTS включать только после проверки HTTPS на обоих доменах.
- Прямой внешний доступ к внутреннему порту приложения закрывать firewall-ом, если используется reverse proxy.

## 4. Автозапуск

Автозапуск можно делать через Windows Task Scheduler, NSSM, PM2 или отдельный service-wrapper. Перед автозапуском нужно вручную убедиться, что production-сборка успешно стартует и сайт открывается.

## 5. Бэкапы

Перед любыми destructive-действиями нужен свежий дамп PostgreSQL и архив проекта без `node_modules`, `.next`, `.env`, uploads и backups.

Минимум перед открытием:

- свежий дамп базы;
- отдельная копия реального `.env` в защищённом месте;
- актуальный код в GitHub;
- проверка восстановления на пустой базе.

## 6. Финальный чеклист

- Репозиторий склонирован с актуальной ветки `main`.
- Реальный `.env` создан только на сервере.
- Новая PostgreSQL-база создана.
- Миграции Prisma применены.
- Production build проходит.
- Production start проходит.
- Регистрация, вход, подтверждение почты, посты, комментарии, реакции, жалобы, админка и Premium-заглушки проверены.
- Оба домена открывают один сайт.
- Юридические страницы и контактные данные проверены владельцем.

## 7. Отдельно

Cloudflare, WAF, CDN, TLS-терминация и дополнительные firewall-правила подключаются отдельным шагом после базового запуска. Не включать proxy-доверие и строгие HTTPS-настройки до проверки всей цепочки запросов.
